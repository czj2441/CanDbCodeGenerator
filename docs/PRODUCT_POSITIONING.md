# CanMatrix Editor —— 产品定位与技术路线决策

> 本文档用于冻结产品边界与技术选型，防止后续开发中的功能蔓延与架构漂移。
> 任何重大偏离需经过显式评审并更新本文档。

---

## 1. 产品定位

### 1.1 一句话定义

CanMatrix Editor 是**面向汽车电子与工业控制工程师的 CAN 矩阵桌面编辑工具**，以 Properties 为主存储格式，解决 DBC 文件在 Git 版本管理中合并冲突严重、diff 不可读的痛点，同时保留与现有 DBC 工具链的完整互通能力。

### 1.2 目标用户

| 用户角色 | 核心诉求 | 使用场景 |
|---------|---------|---------|
| 汽车电子系统工程师 | 多人协作编辑 CAN 矩阵，Git 管理变更 | 日常增删改查信号、报文，提交 Properties 到版本库 |
| 嵌入式软件工程师 | 从矩阵生成代码或配置 | 导出 DBC 给 CANoe/CANalyzer，或供代码生成器消费 |
| 测试/标定工程师 | 审查矩阵变更，追溯历史 | `git diff` 审查信号参数调整，回滚误操作 |
| 开源社区贡献者 | 无商业工具依赖，可二次开发 |  fork 项目，定制企业私有格式扩展 |

### 1.3 核心价值主张

1. **Git-Native 存储**：Properties 格式让每一次信号修改都产生精准、可读的 diff，彻底消除 DBC 行级冲突。
2. **零工具链锁定**：导入现有 DBC → 编辑 → 导出 DBC，与 Vector CANdb++、CANoe 等工具无缝协作。
3. **本地优先**：纯本地运行，数据不出本机，无需账号、云端或网络连接。
4. **双模交互**：Web GUI 用于交互式编辑，CLI/REST API 用于批量处理与自动化流水线集成。

### 1.4 竞争边界（我们不做的事）

| 不做 | 原因 |
|-----|------|
| 实时 CAN 总线监控/记录 | 属于 CANoe/PCAN 领域，与编辑工具职责分离 |
| 代码自动生成（C/Python） | 可由外部工具消费导出的 DBC/Properties 完成，保持单一职责 |
| 多用户并发实时协作 | 复杂度极高，与 Git-based 工作流冲突；Git 已经解决异步协作 |
| 云端托管/SaaS 版本 | 违背本地优先原则，引入运维与安全负担 |
| 完整替代 Vector 全家桶 | 仅聚焦矩阵编辑与格式转换，不做仿真、测试、标定 |

## 2. 技术路线决策

### 2.1 架构模式：轻量级本地 B/S

**决策**：采用 "浏览器前端 + Python HTTP 后端" 的轻量级 B/S 架构，而非传统桌面 GUI（PyQt/Electron）。

**理由**：
- 前端技术栈（Vue 3 + Vite）生态成熟，UI 开发效率远高于 PyQt/PySide
- 浏览器天然支持多标签页，与 Session 隔离模型天然契合
- Python 后端无需打包分发，源码直接运行，跨平台成本低
- 避免 Electron 的体积膨胀（>100MB），整个项目保持轻量（<10MB 不含 node_modules）

**约束**：
- 后端仅限 localhost 绑定，不暴露到公网
- 不使用重量级 Web 框架（Django/FastAPI），标准库 `http.server` + 自定义路由足够

### 2.2 主存储格式：Properties

**决策**：以 Java Properties 为唯一主存储格式，JSON/XML/DBC 为交换格式。

**理由**：
- Properties 的 dotted keys 结构让每条信号独立成块（如 `messages.0x100.signals.RPM`），增删信号只影响该块
- 人类可读性强，审查 PR 时无需工具辅助即可理解变更
- 稀疏输出策略（省略默认值字段）进一步压缩 diff 体积
- 十六进制 ID（`messages.0x123.name`）与 CAN 行业惯例一致

**约束**：
- 内部数据模型（`CanDatabase`）不感知 Properties，序列化层负责格式转换
- 文件扩展名统一为 `.properties`

### 2.3 数据模型：per-message 信号定义

**决策**：信号不设立全局注册表，每个报文拥有独立的信号列表。同名信号在不同报文中允许完全不同属性。

**理由**：
- 与 DBC 语义 1:1 对齐，避免导入/导出时的语义丢失
- 信号作用域限定在报文内，降低模型复杂度（无需全局唯一性校验、无跨报文引用）
- UUID 仅用于运行时的前端-后端定位（`sig.uuid`），不持久化为业务标识

**约束**：
- 不支持"全局信号模板"或"信号继承"概念；如需复用，通过复制粘贴实现
- 信号名称在同一报文内可重复（DBC 允许），但 UI 中不建议

### 2.4 会话与持久化：文件即会话

**决策**：每个编辑会话绑定一个磁盘文件，所有变更即时原子写入；会话通过 `session_id` 索引，支持浏览器崩溃恢复。

**理由**：
- 数据文件是唯一的持久化真相源，内存中的 `CanDatabase` 只是运行时视图
- 原子写入（`.tmp` → `replace`）保证崩溃后文件不会损坏
- `localStorage` 仅存储 `session_id` 和最近会话索引，不缓存业务数据，避免 stale data

**约束**：
- 单用户单设备场景优先，多设备同步通过 Git 解决
- Session 超时（30 分钟）后自动清理内存，但文件保留

### 2.5 并发模型：进程内 RLock

**决策**：`CanDatabase` 使用 `threading.RLock` 保护所有状态变更，`SessionManager` 使用独立锁管理会话生命周期。

**理由**：
- Python 的 GIL + `http.server` 多线程模型下，RLock 足够应对桌面级并发
- 锁层级固定：先 `SessionManager` 锁，后 `CanDatabase` 锁，避免死锁
- 不引入多进程或数据库引擎，保持部署零依赖

**约束**：
- 不承诺多进程并发安全（如同时启动多个后端实例操作同一文件）
- 大文件（>10MB）序列化会阻塞请求线程，但 CAN 矩阵通常 <100KB，可接受

### 2.6 前端状态管理：Pinia + REST API 直读直写

**决策**：前端不维护本地数据库副本，所有变更通过 REST API 同步到后端，成功后更新前端状态。

**理由**：
- 数据真相源在后端，避免前后端状态不一致
- 简化 Undo/Redo 实现（可基于后端快照或操作日志）
- 为后续多标签页编辑（同一 session 多窗口）预留扩展空间

**约束**：
- 批量操作（如导入 500 条信号）需走单独 API，避免 N+1 请求
- 离线编辑不在当前版本支持范围内

## 3. 演进路线（按优先级排序）

### 阶段 1：核心编辑闭环（当前）
- [x] 报文/信号的 CRUD
- [x] Properties/JSON/DBC 导入导出
- [x] 会话管理与自动持久化
- [x] 多标签页隔离（sessionStorage + localStorage 索引）
- [x] 数据模型线程安全加固

### 阶段 2：工程化体验
- [ ] 批量导入/导出（Excel/CSV 交换）
- [ ] 信号搜索与筛选
- [x] 报文/信号复制粘贴
- [ ] 拖拽排序
- [ ] 变更历史面板（基于文件系统 mtime 或简单操作日志）
- [x] 前端 Undo/Redo

### 阶段 3：生态系统集成
- [ ] CLI 工具完善：`canmatrix edit`、`canmatrix convert`、`canmatrix validate`
- [ ] Git diff 渲染器（将 Properties diff 渲染为信号变更表格，供 CI 审查）
- [ ] VS Code 插件（Properties 语法高亮 + 信号预览）
- [ ] 与 cantools 的深度融合（利用 cantools 的代码生成能力）

### 阶段 4：企业级扩展（待定）
- [ ] 信号库/报文模板（预定义常用信号集）
- [ ] 矩阵校验规则（ID 冲突、位域重叠、名称规范）
- [ ] 多语言矩阵同步（同一物理信号在不同语言项目中的映射管理）

> **原则**：阶段 2 完成前，不启动阶段 3；阶段 3 的每一项都必须保持零额外运行时依赖。

## 4. 决策冻结与变更流程

本文档中的决策在阶段 2 完成前保持冻结。如需变更：

1. 在 Issue/PR 中明确说明偏离的决策项
2. 提供技术对比与成本分析
3. 经维护者评审通过后更新本文档
4. 重大架构变更（如引入数据库、改为 Electron、增加实时协作）需单独召开技术评审会
